
# 项目总结
### 图像数据
各种病虫害图像的数量是不平衡的，验证性能时需要使用多种指标，准确率、查准率、召回率和 F1 指数等，划分数据集的比例一般是训练：验证：测试= 6:2:2、7:2:1 或 8:1:1 ，需要确保训练的图像足够。如果图像数据较少，建议使用预训练模型直接训练，必要时可冻结除了分类层以外的所有层，只训练分类层的权重和偏置参数。

### 模型训练
在训练脚本中，各种训练参数最好在主函数中以参数的形式定义，便于更改和调试，比如数据路径、batchsize、epoch 等。 

关于早停机制，之前的实现方式有错误，保存的不是验证集准确率最高的那个 epoch 对应的模型数据，而是当前训练的 epoch 对应的模型数据。所以，如果当前 epoch 的验证准确率比之前所有 epoch 的最高验证准确率高，就要保存当前 epoch 对应的权重，如果后面的验证准确率连续一定 的轮次不再上升，就加载之前保存的验证准确率最高的模型权重进行保存，才能得到所谓的验证准确率最高的 epoch 对应的模型数据。对于占用内存较大的模型，可能会保存较多的版本，需要较大的内存空间。  

使用 TensorBoard 保存每个 epoch 对应的数据文件，进行可视化，方便展示。  

### 模型测试
由于生成的图像标签名字比较长，没有重新修改为较短的名字，所以混淆矩阵在可视化时比较混乱，之后可将标签更改为较短的首字母组合。  
在 gpu 显存有限的情况下，多次执行容易导致 cuda out of memory，建议在执行开始前添加清空 cuda 缓存的代码。
```python
# 清空 cuda 缓存
torch.cuda.empty_cache()
```

验证和测试阶段都不需要计算梯度，一般添加下面代码
```python
model.eval()
with torch.no_grad():
    pass # 推理过程
```
### 模型部署
部署到 jetson-nano-b01 时，模型转换一般不会出现问题，主要是开发板的内存空间紧张，容易超出。所以不必要的东西尽量不安装（不用桌面模式，使用终端模式，不安装如 vscode 等的集成开发环境）代码在笔记本等支持开发的环境中写好，然后远程连接传输到开发板进行测试即可。  
最好安装 jtop：`sudo pip3 install jetson-stats`，监控 GPU/CPU 使用率
